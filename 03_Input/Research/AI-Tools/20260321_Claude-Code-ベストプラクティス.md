# Claude Codeのベストプラクティス

## 調査概要
- **調査日**: 2026-03-21
- **テーマ**: Claude Codeを効果的に活用するためのベストプラクティス
- **情報源**: 公式ドキュメント、実践知見、既存ノートから整理

## 1. CLAUDE.md設計のベストプラクティス

### 基本原則
- プロジェクトルートに `CLAUDE.md` を配置
- 重要な情報を簡潔に記述（LLMが読むことを意識）
- コマンド定義は `.claude/commands/` に分離
- 条件付きルールは `.claude/rules/` に分離

### CLAUDE.mdに書くべき内容
- プロジェクトの概要・フォルダ構成
- 命名規則・コーディング規約
- 利用可能なコマンド一覧
- ファイル配置ルール
- 作業範囲の制限

### アンチパターン
- CLAUDE.mdに全ルールを詰め込む → ファイルが肥大化しトークン消費増大
- 曖昧な指示（「適切に処理して」）→ 具体的な条件・閾値を記述すべき
- ネストされたフォルダ構造の説明不足 → 明示的なパス例を提示

## 2. カスタムコマンド設計

### 推奨パターン
- 1コマンド = 1目的（Single Responsibility）
- 引数は `$ARGUMENTS` で受け取り
- 手順をステップバイステップで記述
- 出力先パスを明示的に指定

### ファイル構成
```
.claude/commands/
├── daily.md          # 日次ルーチン
├── memo-to-inbox.md  # メモ整理
├── research.md       # リサーチ
├── blog-generate.md  # ブログ記事生成
└── weekly-review.md  # 週次レビュー
```

## 3. .claude/rules/ の活用

### 条件付きルール
- ファイルパス・パターンに基づいて自動適用されるルール
- ブログ記事のスラッグルール、ファクトチェック要件など
- コマンドとの違い: コマンドは明示的発火、ルールは暗黙的適用

## 4. Hooksの活用

### 利用可能なフック
- `beforeSubmitPrompt` — プロンプト送信前
- `afterFileEdit` — ファイル編集後
- `stop` — セッション終了時

### 活用例
- 編集ログの自動記録
- セッションの振り返り生成
- プロンプトと意図の記録

## 5. サブエージェント（Agent Tool）

### 並列処理パターン
- 独立したタスクは並列実行（例: 3チャネル同時コンテンツ生成）
- ファイル所有権ルールで衝突防止
- 同一ファイルを複数エージェントが編集しないこと

### 逐次処理すべきパターン
- パイプライン（各ステップが前の出力に依存）
- 同一ファイルへの書込み
- 2分以内の単体コマンド

## 6. セカンドブレイン統合

### PARA法との統合
- Projects / Areas / Resources / Archive に対応するフォルダ構成
- CLAUDE.mdにファイル配置判断フローを定義
- タグ規則・リンク戦略を明記

### 効果的なワークフロー
1. メモ取得 → 00_Memo/
2. 自動整理 → /memo-to-inbox でスコアリング・振り分け
3. ナレッジ蓄積 → 04_Memory/ にMOC付きで保存
4. アウトプット → 05_Output/ でプロジェクト・エリア管理
